# Victor AI 面试助手 - 核心实现规范

## 1. LLM适配层实现

### 1.1 Spring AI统一接口

LLM适配层直接使用 Spring AI 框架提供的统一接口，无需自行抽象协议适配层。Spring AI 已内置对 OpenAI、Anthropic Claude、Ollama 等主流模型的支持。

**核心组件：**

- **ChatClient**：Spring AI 提供的统一聊天客户端，支持同步和流式调用。业务层通过 `ChatClient` 发送请求，底层自动路由到对应的模型提供商。

- **ChatModel**：聊天模型接口，实现包括 `OpenAiChatModel`、`AnthropicChatModel`、`OllamaChatModel` 等。

- **Prompt / Message**：统一的提示词和消息对象，支持 SystemMessage、UserMessage、AssistantMessage 等角色。

- **ChatResponse**：统一响应对象，包含生成内容、token使用情况等。

### 1.2 多模型实例配置

通过 Spring AI 的配置机制管理多个模型实例，每个实例绑定不同的提供商和参数。

**配置方式：**

1. 在 `application.yml` 中配置各模型提供商的连接信息（API Key、端点、模型名称等）。

2. 使用 `@Bean` 注册多个 `ChatModel` 实例，通过 `@Qualifier` 区分。

3. 业务层通过 `agent_llm_config` 表中的配置动态选择对应的 `ChatModel` 实例。

**支持的模型提供商**：
- OpenAI（GPT系列、兼容OpenAI协议的模型如通义千问）
- Anthropic Claude
- Ollama（本地模型）

### 1.3 流式响应

Spring AI 原生支持流式响应，通过 `stream()` 方法返回 `Flux<ChatResponse>`，配合 WebSocket 实现实时字幕推送。

## 2. 资料召回实现

### 2.1 召回策略

系统不采用独立的知识库模块，而是基于简历（res_resume）和岗位（res_job）中的结构化数据提取关键词，在题库、经历库等资料表中进行检索。

**召回流程：**

1. **关键词提取**：从简历的 `parsed_content`（结构化解析内容）和岗位的 `required_skills`、`description` 中提取关键技能词、领域词、岗位要求词。

2. **资料检索**：使用提取的关键词在以下资料源中检索：
   - **题库（res_question）**：按标签（tags）、难度、类型匹配面试题
   - **经历库（res_experience）**：按技能（skills）匹配用户经历
   - **岗位库（res_job）**：按领域（domains）、技能要求匹配相关岗位

3. **结果排序**：根据匹配度、相关性对召回结果排序，控制每类资料的召回数量上限。

4. **用户确认**：召回结果写入 `interview_config.recall_items` 工作区，用户可审阅、删除、手动添加，确认后冻结。

### 2.2 召回数据引用

召回结果采用引用方式存储在 `recall_items` JSON字段中，不冗余保存原始资料全文。每条引用包含：
- `source_type`：来源类型（QUESTION、JOB、RESUME、EXPERIENCE）
- `source_id`：来源资料ID
- `recall_method`：召回方式（AUTO自动/MANUAL手动）
- `recall_score`：召回匹配分数
- `fragments`：引用片段（支持字段级、范围级引用）

**设计要点：**

- 原始资料被历史面试引用时，删除或修改前需提示用户影响范围。
- 原始资料通过软删除隐藏，历史面试回看时无法解引用则展示"资料已删除"提示。

## 3. Agent引擎实现

### 3.1 Agent执行引擎

AgentEngine负责执行Agent的逻辑，包括单次执行、流式执行和Loop执行。

**单次执行流程：**

1. 获取Agent配置的模型实例。

2. 构建Prompt：组合System Prompt、注入上下文变量、添加对话历史。

3. 调用LLM：使用模型实例发送请求到LLM。

4. 解析响应：检查是否需要调用工具（通过特定格式识别tool call）。

5. 如果需要工具：
   - 执行工具，获取结果。
   - 将工具结果加入上下文。
   - 重新调用LLM，让Agent基于工具结果继续推理。

6. 返回最终结果，包含内容和token使用情况。

**流式执行**：与单次执行类似，但使用streamChat()接收流式响应，逐步返回给前端，降低用户等待时间。

**Loop执行**：循环执行Agent直到满足终止条件。

1. 初始化循环计数器。

2. WHILE 不满足终止条件 AND 未达到最大循环次数：
   - 执行一次Agent。
   - 更新上下文（添加本轮结果）。
   - 检查终止条件（如Loop Agent的信息足够判断）。

3. 返回最终累积的结果。

**应用场景**：Loop Agent检索、多轮追问等需要反复迭代的场景。

### 3.2 Agent工具定义

Tool接口定义Agent可调用的工具。

**工具接口方法：**

- `getName()`：工具名称，如"search"、"question_query"。

- `getDescription()`：工具描述，帮助LLM理解何时使用该工具。

- `getParametersSchema()`：参数JSON Schema，定义工具接受的参数结构。

- `execute(arguments)`：执行工具，接收参数字典，返回执行结果。

**内置工具示例：**

- **SearchTool（搜索工具）**：根据简历和岗位提取的关键词，在题库、经历库等资料表中检索相关信息，接受query和topK参数。

- **QuestionTool（题库查询工具）**：从题库中按条件查询面试题。

- **ResumeQueryTool（简历查询工具）**：查询用户简历的结构化数据。

**工具注册**：系统启动时将工具注册到ToolRegistry，Agent执行时按需调用。

### 3.3 Agent调度器

AgentOrchestrator协调多个Agent的执行，支持并行和串行两种模式。

**并行执行（executeTeamParallel）：**

1. 获取Agent团队配置和所有成员Agent。

2. 使用CompletableFuture并行执行所有Agent。

3. 收集所有结果，返回结果列表。

**适用场景**：多维度评估（技术评估、行为评估、领域评估同时进行）。

**串行执行（executeTeamSequential）：**

1. 获取Agent团队配置，按priority排序成员。

2. 依次执行每个Agent，前一个结果传递给下一个作为上下文。

3. 返回结果列表。

**适用场景**：出题Agent → 追问Agent的顺序执行。

## 4. 语音服务实现

### 4.1 ASR服务接口

AsrAdapter定义语音识别适配器接口。

**核心方法：**

- `getProviderName()`：返回服务商名称（如QWEN、DOUBAO）。

- `transcribe(audioData, config)`：同步识别，接收音频字节数组和配置，返回识别结果（文本、置信度、是否最终结果）。

- `streamTranscribe(audioStream, config)`：流式识别，接收音频流，返回识别结果流，支持实时字幕。

**ASR配置参数：**

- language：语言（如zh-CN、en-US）
- format：音频格式（wav、mp3等）
- sampleRate：采样率

### 4.2 TTS服务接口

TtsAdapter定义语音合成适配器接口。

**核心方法：**

- `getProviderName()`：返回服务商名称。

- `synthesize(text, config)`：同步合成，接收文本和配置，返回音频字节数组。

- `streamSynthesize(text, config)`：流式合成，接收文本，返回音频流，降低首字延迟。

**TTS配置参数：**

- voice：音色（如zhichu、xiaoyun）
- speed：语速（0.5-2.0）
- pitch：音调
- format：输出格式（wav、mp3等）

### 4.3 语音服务统一入口

VoiceService作为语音服务的统一入口，根据配置路由到对应的ASR/TTS适配器。

**工作流程：**

1. 根据configId获取语音服务配置（包含provider、apiEndpoint、apiKey等）。

2. 从适配器映射表中找到对应的ASR或TTS适配器。

3. 构建配置对象（language、format、voice等）。

4. 调用适配器的transcribe()或synthesize()方法。

5. 返回识别结果或音频数据。

**流式处理**：面试场景中，ASR和TTS都使用流式接口，配合WebSocket实现实时字幕和低延迟语音播放。

## 5. 面试引擎实现

### 5.1 面试引擎核心

InterviewEngine管理面试的全生命周期。

**核心功能：**

**生成面试题（generateQuestion）：**

1. 获取当前轮次配置、已问过题目、岗位要求、简历信息。

2. 构建AgentContext，注入round、job、resume、askedQuestions等变量。

3. 获取出题Agent配置。

4. 调用AgentEngine执行出题Agent，生成新题目。

5. 确保题目不与已问过的重复。

6. 解析Agent返回的内容为Question对象。

**评估回答并决定下一步（evaluateAnswer）：**

1. 获取当前题目、用户回答、岗位要求、简历信息。

2. 构建AgentContext，注入question、answer、job、resume等变量。

3. 调用评估Agent评估回答质量。

4. 判断是否需要追问：
   - 回答深度是否足够
   - 是否涉及关键知识点
   - 岗位要求的核心技能是否覆盖

5. 返回下一步动作：
   - FOLLOW_UP：需要追问
   - NEXT_QUESTION：进入下一题
   - END：结束面试

**生成追问（generateFollowUp）：**

1. 调用追问Agent，传入当前题目和用户回答。

2. Agent根据回答的不足之处生成针对性追问。

3. 返回追问问题文本。

### 5.2 面试会话管理

**面试开始：**

1. 加载面试配置（interview_config）。

2. 创建面试会话（interview_session），状态设为IN_PROGRESS。

3. 将配置状态更新为ARCHIVED（锁定配置）。

4. 获取第一道题目（interview_question）。

5. 初始化Agent团队。

6. 返回会话ID和WebSocket地址给前端。

**面试过程：**

1. 前端通过WebSocket发送用户回答（文字、语音、图片、代码等）。

2. 后端保存对话回合（interview_turn），记录speaker、content、attachments等。

3. 如果是语音模式，调用ASR转文字。

4. 调用面试引擎评估回答，决定下一步（追问/下一题/结束）。

5. 生成AI回复，可能包含文字、代码、Mermaid图等。

6. 如果是语音模式，调用TTS转语音。

7. 通过WebSocket推送AI回复给前端。

8. 实时更新current_question_id指针。

**暂停/恢复：**

- 暂停：将会话状态更新为PAUSED，记录paused_at时间。

- 恢复：将会话状态更新为IN_PROGRESS，AI提示"我们继续上次的面试"。

**结束面试：**

1. 将会话状态更新为COMPLETED，记录completed_at时间。

2. 创建面试报告（interview_report），状态设为PENDING。

3. 触发异步评估流程。

### 5.3 面试报告生成

**评估流程：**

1. 获取面试会话的所有对话回合（interview_turn）。

2. 获取面试题目列表（interview_question）。

3. 构建AgentContext，注入完整面试记录。

4. 并行执行多个评估Agent：
   - 技术评估Agent：评估技术能力、代码质量
   - 行为评估Agent：评估沟通能力、软技能
   - 领域专家Agent：评估特定领域知识深度

5. 综合评估Agent汇总各维度评分，生成：
   - overallScore：总分（0-100）
   - dimensionScores：各维度评分（技术能力、项目经验、沟通表达等）
   - perQuestionEvaluation：逐题评估
   - summary：综合总结
   - strengths：优势项及证据
   - weaknesses：劣势项及改进建议
   - suggestions：针对性的学习建议

6. 更新报告状态为COMPLETED，记录generated_at时间。

**失败重试**：如果评估失败，记录evaluation_error和evaluation_retry_count，支持自动重试。

## 6. 开放接入实现

### 6.1 开放API Key管理

OpenApiKeyService管理用于外部采集程序的API Key。

**Key属性：**

- name：Key名称
- apiKey：明文API Key（唯一）
- scopes：权限范围数组（如IMPORT_JOB、IMPORT_QUESTION、IMPORT_RESUME、IMPORT_EXPERIENCE）
- default_ingest_status：默认导入状态（ACTIVE或PENDING_REVIEW）
- status：启用/禁用状态
- expires_at：过期时间（可选）
- last_used_at：最近使用时间

**鉴权流程：**

1. 外部程序在请求头中携带Authorization: Bearer {api_key}。

2. 后端拦截器提取api_key，查询open_api_key表验证有效性。

3. 检查status是否为ENABLED，expires_at是否过期。

4. 检查请求的接口是否在scopes权限范围内。

5. 验证通过后，将user_id和api_key_id存入上下文，供后续使用。

### 6.2 开放数据导入

OpenImportService提供按资料类型分开的导入接口。

**导入接口：**

- `/open-api/v1/jobs/import`：导入岗位
- `/open-api/v1/questions/import`：导入题目
- `/open-api/v1/resumes/import`：导入简历
- `/open-api/v1/experiences/import`：导入经历

**导入流程：**

1. 接收外部程序的POST请求，携带items数组。

2. 遍历每个item，包含externalId、sourceUri、rawPayload和标准化字段。

3. 根据default_ingest_status设置ingest_status（ACTIVE或PENDING_REVIEW）。

4. 填充来源追踪字段：source_type=OPEN_API、source_api_key_id、source_uri、external_id、raw_payload。

5. 插入对应的资料表（res_job、res_question、res_resume、res_experience）。

6. 返回导入结果（成功数量、失败数量、错误详情）。

**外部采集程序**：系统不管理外部程序的运行方式，可以是浏览器扩展、Shell脚本、远程服务或内部程序，只要调用开放导入接口即可。

### 6.3 待审核资料管理

当API Key的default_ingest_status为PENDING_REVIEW时，导入的资料不会参与AI召回。

**审核流程：**

1. 用户在资料列表中筛选ingest_status=PENDING_REVIEW的资料。

2. 查看资料详情，确认质量。

3. 执行确认操作：将ingest_status更新为ACTIVE，资料可用于面试召回和题目生成。

4. 或执行拒绝操作：将ingest_status更新为REJECTED，资料不参与召回。

**设计要点：**

- AI召回和面试题生成默认只使用ACTIVE资料。
- 原始资料被历史面试引用时，修改或删除前需要提示影响范围。
- 用户确认后允许操作，回看时如果无法正常解引用，展示资料已删除或已变更的提示。
